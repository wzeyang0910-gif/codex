# 原研医疗内部获客平台首版设计规格

日期：2026-07-10

## 1. 项目目标

搭建一个仅供常州原研医疗器械有限公司内部使用的获客网站。业务员输入或选择原研现有产品、目标地区和客户类型后，系统自动完成市场调研、潜在客户搜索、公司背景调查、关键联系人挖掘、邮箱验证、客户分级和个性化开发信生成。

首版目标不是生成每日 Excel 或 Word 文件，而是在网站内直接展示、维护和编辑客户信息，形成可持续更新的内部客户库和跟进系统。

## 2. 首版范围

首版面向 1-5 名内部业务员使用，采用本地部署优先，后续再考虑云端部署。

必须包含：

- 登录与角色权限：管理员、业务员。
- 产品选择：固定使用原研产品库，可多选产品，也可补充应用场景或关键词。
- 地区选择：支持选择中东、日本、台湾等大区，也可细化到国家或地区。
- 客户类型选择：进口商、分销商、品牌商、零售商、电商、运动用品渠道、医疗机构供应链等。
- 每次任务交付 5 家有效客户。
- 每名业务员每天最多获取 30 家客户。
- 只交付 A 类和 B 类客户，不交付 C 类客户。
- 每家有效客户必须至少有 1 个关键负责人或相关负责人的个人工作邮箱。
- 邮箱需要通过 Prospeo、Hunter、Apify、ContactOut 等渠道组合验证，并记录验证状态和来源。
- 客户信息、联系人、开发信、跟进状态都在网站内展示和编辑。

首版不包含：

- Gmail、Outlook 自动发送邮件。
- Excel/Word 每日文件输出。
- 面向外部用户的注册、付费、订阅或公开 SaaS 功能。
- 对外开放的通用产品获客平台。

## 3. 用户角色

管理员：

- 创建和管理业务员账号。
- 查看全部客户、全部任务和全部跟进记录。
- 维护原研产品库、公司优势、认证信息、产品目录 PDF。
- 查看 API 调用状态、余额、失败记录、任务耗时和获客成本。
- 管理去重规则、评分规则、每日额度和邮箱验证规则。

业务员：

- 创建获客任务。
- 查看自己任务产生的客户。
- 编辑自己负责客户的客户信息、联系人、备注和跟进状态。
- 查看系统生成的开发信并自行复制发送。
- 更新客户阶段：未联系、已发送、已回复、已询价、已成交、无意向。

客户归属规则：

- 业务员默认只能看到自己的客户。
- 管理员可查看所有客户。
- 系统按公司名称、官网域名、品牌名称和国家/地区做全局查重，避免不同业务员重复开发同一客户。

## 4. 产品与目标市场逻辑

系统内置原研产品库，不要求用户每次重新输入公司资料。

核心产品范围包括：

- 医用胶带系列：PE 胶带、无纺布胶带、丝绸胶带、棉布胶带、无纺布敷料卷、PU 敷料卷、硅凝胶胶带等。
- 肌内效贴系列：棉布肌贴、人造丝肌贴、印花肌贴、预切肌贴等。
- 自粘绷带系列：棉布自粘绷带、无纺布自粘绷带、印花自粘绷带等。
- 运动胶带系列：运动胶带、彩色运动胶带、曲棍球胶带、Underwrap 等。
- 粘性绷带和弹性绷带系列：EAB、轻弹绷带、PBT 绷带、POP 石膏绷带、石膏衬垫等。
- 创口贴、敷料贴、痘痘贴、防水贴、导管固定贴、疤痕贴等。

搜索客户时不能只搜索产品名，必须组合以下关键词：

- 产品词：kinesiology tape, cohesive bandage, surgical tape, wound dressing, plaster bandage 等。
- 场景词：sports medicine, physiotherapy, orthopedic, first aid, wound care, pharmacy supply, rehabilitation 等。
- 客户类型词：importer, distributor, wholesaler, medical supplier, sports tape brand, pharmacy chain, e-commerce seller 等。
- 竞品词：当地同类品牌、进口商名、分销商名、产品系列名。
- 海关词：import, shipment, HS code, supplier, buyer, China export 等。

系统需要扩大目标客户范围，不局限于医疗行业，也覆盖运动康复、体育用品、药房渠道、电商渠道、急救用品、理疗康复、骨科耗材等 B 端客户。C 端客户可作为补充，但首版优先 B 端。

## 5. 获客任务流程

1. 业务员进入任务页。
2. 选择产品、目标地区、国家/地区、客户类型和开发信语言。
3. 系统显示今日剩余额度，例如“今日还可获取 25 家客户”。
4. 用户提交任务。
5. 系统开始异步执行任务，目标是在 10 分钟内完成 5 家有效客户。
6. 任务进度页展示搜索状态，例如已检索公司数、进入深度调查公司数、已验证联系人数量、已淘汰公司数量。
7. 系统持续替换无效候选客户，直到获得 5 家 A/B 客户，或达到首轮时间限制。
8. 用户在网站结果页查看客户详情、联系人、开发信和推荐开发策略。
9. 用户可将客户加入自己的开发客户库，并维护后续跟进。

如果 10 分钟内不足 5 家有效客户，系统展示已完成的有效客户，同时后台继续补齐。未补齐的客户不扣额度。

## 6. 有效客户标准

客户必须同时满足：

- 是真实存在的公司，官网、社媒、目录平台、海关数据或第三方数据库中至少有两个可追溯来源。
- 与原研至少一个产品系列存在明确业务匹配。
- 具备进口、分销、批发、品牌采购、线上销售、线下渠道或机构采购需求中的至少一种证据。
- 找到至少一位老板、企业负责人、采购负责人、运营负责人、产品负责人或相关业务负责人。
- 至少一个个人工作邮箱通过验证，或标记为 accept-all 但有足够身份证据支持。

无效客户包括：

- 只有公共邮箱且找不到负责人个人邮箱。
- 公司规模过大且供应链明显不适配原研当前阶段。
- 业务与原研产品关系弱。
- 无法确认真实存在或公司信息严重冲突。
- 只适合 C 类低优先级开发的客户。

## 7. 联系人与邮箱规则

联系人优先级：

1. 老板、创始人、CEO、Managing Director、General Manager。
2. Procurement Manager、Purchasing Manager、Sourcing Manager。
3. Product Manager、Category Manager、Business Development Manager。
4. Sales Director、Operations Manager、Import Manager。

每家公司尽量输出多个联系人。每个联系人需要记录：

- 姓名。
- 职位。
- 邮箱。
- 邮箱验证状态：valid、accept-all、risky、invalid、unknown。
- 邮箱来源：Prospeo、Hunter、Apify、ContactOut、公司官网、社媒、目录平台等。
- 联系人来源链接或证据说明。
- 是否推荐优先联系。

规则：

- valid 邮箱优先。
- accept-all 邮箱可以进入 B 类，但必须醒目标注风险。
- invalid 邮箱不得作为有效联系方式。
- 找不到个人邮箱的公司直接淘汰并替换。
- 不得编造联系人、职位、邮箱或验证结果。

## 8. 客户分级

只交付 A 类和 B 类客户。

A 类客户：

- 产品高度匹配。
- 有进口、采购、分销或品牌经营证据。
- 找到老板或采购相关负责人。
- 至少一个 valid 个人工作邮箱。
- 公司规模为中大型、中型或中小型，适合工厂端开发。
- 有明确合作切入点，例如 OEM/ODM、价格、认证、稳定交期、工厂验厂、产品线补充。

B 类客户：

- 产品匹配较强。
- 有采购或渠道需求证据，但证据不如 A 类完整。
- 找到关键联系人。
- 邮箱为 valid 或 accept-all。
- 有开发价值，但需要更谨慎跟进。

C 类客户：

- 不在网站结果中交付。
- 可保留在后台淘汰记录中，用于分析为什么被排除。

评分维度：

- 产品匹配度。
- 客户规模适配度。
- 采购或进口需求证据。
- 联系人质量。
- 邮箱验证质量。
- 市场渠道匹配度。
- 与原研优势的合作切入点。

## 9. 市场调研与竞品分析

每个地区任务开始前，系统需要先生成目标市场分析，包括：

- 当地医疗耗材、运动康复、绷带胶带相关市场需求。
- 当地客户采购习惯。
- 当地客户更关注的合作点，例如认证、价格、交期、独家代理、OEM 包装、付款方式、稳定供应。
- 主要竞品和同类产品渠道。
- 推荐优先开发的客户类型。
- 本次任务使用的关键词策略。

市场分析不要求写成长报告，但必须影响候选客户选择和开发信内容。

## 10. 客户详情页

每家客户在网站中应展示以下模块：

- 基础信息：公司名称、国家/地区、城市、官网、主营业务、客户类型、规模判断。
- 来源证据：官网、社媒、目录平台、海关数据、中国出口数据、第三方数据库等。
- 需求证据：进口记录、产品页面、品牌目录、采购岗位、分销渠道、社媒销售内容等。
- 匹配产品：推荐开发的原研产品系列及理由。
- 竞品或现有供应链线索：如可查到则展示。
- 联系人：多个关键联系人、职位、邮箱、验证状态、来源。
- 客户等级：A 或 B，并展示评分组成。
- 推荐开发策略：合作切入点、首封邮件重点、后续跟进建议。
- 个性化开发信：结合客户业务背景、原研产品优势、认证、工厂直供、可验厂和目录附件。
- 跟进状态：未联系、已发送、已回复、已询价、已成交、无意向。
- 用户备注：业务员可编辑。

客户信息需要允许用户编辑。系统保留原始采集信息和用户编辑后的信息，避免用户修改后丢失证据来源。

## 11. 开发信生成规则

开发信必须专业、委婉、真诚、有针对性，不能只是套模板换公司名。

每封开发信需要体现：

- 对客户业务的具体理解。
- 该客户可能适合的原研产品。
- 原研是一家位于中国常州的生产工厂。
- 可提供 OEM/ODM、工厂直供、稳定质量、价格优势、交期支持。
- 具备 ISO13485、CE、FDA 等资质。
- 可安排验厂、参观工厂和生产流程。
- 邮件中可提及会附上 PDF 产品目录，但系统首版不负责自动发送附件。
- 语气客气，不强推，鼓励客户提出当前采购需求。

默认语言为英文，用户可选择其他语言。

开发信约 300-500 英文词，可根据客户情况调整，不机械限制字数。

## 12. 系统架构

首版采用模块化单体架构，方便本地部署和快速迭代。

前端：

- 登录页。
- 任务创建页。
- 任务进度页。
- 客户结果页。
- 客户库页。
- 客户详情与编辑页。
- 跟进管理页。
- 管理员后台。

后端：

- 用户与权限服务。
- 产品知识库服务。
- 获客任务服务。
- 客户公司库服务。
- 联系人和邮箱验证服务。
- 证据保存服务。
- 客户评分服务。
- 开发信生成服务。
- 跟进记录服务。
- API 调用日志和成本统计服务。

异步任务：

- 任务提交后进入后台队列。
- 市场调研、候选公司搜索、深度调查、联系人查找、邮箱验证、评分、开发信生成可分阶段执行。
- 任务关闭页面后仍可继续运行。

数据存储：

- 用户表。
- 产品表。
- 公司知识库表。
- 客户公司表。
- 联系人表。
- 邮箱验证表。
- 证据表。
- 获客任务表。
- 客户评分表。
- 开发信表。
- 跟进记录表。
- API 调用日志表。

建议技术：

- Next.js + TypeScript。
- PostgreSQL。
- Redis + BullMQ 或同类任务队列。
- 本地文件存储产品目录 PDF。
- API Key 仅存服务端环境变量，不展示给业务员。

## 13. 外部数据源

系统通过适配器统一接入外部数据源，便于后续替换或增加供应商。

首版优先数据源：

- 搜索引擎结果。
- 公司官网。
- Facebook、Instagram 等社媒页面。
- 行业目录和 B2B 平台。
- 中国出口海关数据或相关公开/第三方数据。
- Prospeo。
- Hunter。
- Apify。
- ContactOut。
- Yokdo 或其他公司联系人查询工具。

所有外部结论必须保存来源。AI 只能基于来源做整理、归纳和判断，不允许凭空生成事实。

## 14. 异常处理

- 单个搜索渠道失败时，自动切换备用渠道。
- API 余额不足时，停止调用该接口，并在管理员后台显示告警。
- 公司信息冲突时，降低可信度，并寻找第二来源。
- 找不到关键联系人个人邮箱时，淘汰该公司。
- 邮箱为 accept-all 时，只能作为 B 类客户，并标注风险。
- 任务时间超过 10 分钟但不足 5 家有效客户时，展示已完成客户，后台继续补齐。
- 重复公司命中历史客户库时，不交付给新业务员。
- 开发信生成失败时，允许重新生成，不影响客户基础数据。

## 15. 管理员后台

管理员后台包含：

- 用户管理。
- 产品库管理。
- 公司资料和卖点管理。
- 产品目录 PDF 上传和替换。
- 全局客户库。
- 任务监控。
- API 状态和余额。
- API 调用成本和失败记录。
- 客户质量统计。
- 邮箱验证结果统计。
- 淘汰客户原因统计。
- 每名业务员额度和任务记录。
- 去重规则和评分规则设置。

## 16. 首版验收标准

首版完成后，需要满足：

- 业务员可以登录并创建获客任务。
- 单次任务目标 10 分钟内完成 5 家有效客户。
- 每名业务员每天最多获取 30 家客户。
- 每家公司至少有 1 位关键负责人或相关负责人的个人工作邮箱。
- 每个邮箱有验证状态和来源。
- 每家公司有可追溯来源和需求证据。
- 只交付 A/B 类客户。
- 不同业务员之间全局查重。
- 客户结果直接在网站中展示。
- 用户可以在网站内编辑客户信息、联系人、备注和跟进状态。
- 每家公司有针对性的开发信。
- 管理员可以查看任务、客户、用户、API 调用和质量统计。
- 系统不编造数据，无法确认的信息必须标记为未知或淘汰。

## 17. 后续可扩展功能

后续可以增加：

- Excel/Word 导出。
- 邮件草稿箱集成。
- Gmail/Outlook 自动发送前确认。
- 自动跟进提醒。
- 客户回复识别。
- 成交转化统计。
- 多公司、多产品 SaaS 化。
- 云端部署和团队协作。

这些功能不进入首版，避免范围过大影响核心获客质量验证。
